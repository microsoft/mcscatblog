## The Identity Crisis

You've built a Copilot Studio agent and [promoted it across environments](https://learn.microsoft.com/en-us/microsoft-copilot-studio/authoring-solutions-import-export) via solution export/import: Dev, Test, Prod. You deploy each one to Teams. Your developers and testers open Microsoft Teams and see three identical blue icons, all named "Support Agent."

Which one is production? Which one is the dev instance they're supposed to be testing?

Nobody knows.

This isn't a discoverability problem, it's an ALM (Application Lifecycle Management) problem. Copilot Studio's 1-click deploy to Teams generates the same name, same icon, same description, and same manifest regardless of which environment the agent lives in.

The solution? Environment-based deployment strategies that differentiate your agents and give you precise control over distribution, auto-installation, and pinning in the Teams sidebar.

> If you haven't optimized your agent for Teams yet, start with [Deploying Copilot Studio Agents in Teams (Because Test Chat Was Too Easy)]({% post_url 2026-04-07-copilot-studio-teams-agent-patterns %}) for production patterns covering reinstalls, inactivity, error handling, and diagnostic cards.
{: .prompt-info }

## Why Standard Deployment Falls Short

Copilot Studio's [1-click deploy to Teams](https://learn.microsoft.com/microsoft-copilot-studio/publication-add-bot-to-microsoft-teams) works great for single-environment scenarios. Click "Add to Teams," publish, and you're done. But it doesn't account for:

- **Environment differentiation** - DEV, TEST, and PROD agents look identical
- **Controlled distribution** - No granular control over who can access what
- **Professional rollouts** - No auto-install or pinning capabilities
- **Governance requirements** - Limited approval workflows

The root cause? The [Teams app manifest](https://learn.microsoft.com/microsoftteams/platform/resources/schema/manifest-schema) generated by Copilot Studio is environment-agnostic. It doesn't know about DEV, TEST, or PROD.

Different environments have different needs:

| Environment | User Base | Stability | Deployment Method | Auto-Install | Pinning |
|-------------|-----------|-----------|-------------------|--------------|---------|
| **DEV** | Individual makers | Unstable, frequent changes | Direct install | No | No |
| **TEST** | QA team, stakeholders | Stable, controlled | Maker submission | Optional | No |
| **PROD** | All users | Stable, audited | Admin-led | **Yes** | **Yes** |

This table drives our deployment strategy. Let's dive into the three distinct deployment paths.

## Understanding the Three Deployment Paths

There are **three distinct deployment paths** for Copilot Studio agents in Teams, depending on who performs the action (maker vs admin) and the level of governance required.

### Deployment Path Comparison

| Aspect | Option 1: Direct Install | Option 2: Maker Submission | Option 3: Admin-Led |
|--------|-------------------------|---------------------------|---------------------|
| **Best For** | Development, testing | Controlled rollout, QA | Production, maximum control |
| **Governance** | Low | Medium | High |
| **Speed** | Fastest | Moderate | Slower |
| **Approval** | None | Admin required | Admin controlled |
| **Initiator** | User or maker | Maker | Admin |
| **Auto-Install** | No | No | **Yes** |
| **Pinning** | No | No | **Yes** |
| **Manifest Control** | Full (sideloading) | Limited | Full |

Let's examine each path in detail.

### Option 1: Direct Install (User-Level)

> **Best for:** Development, testing, limited distribution
> **Governance Level:** Low • **Speed:** Fastest • **Approval:** None
{: .prompt-info }

Users can install the agent **without admin involvement**, in two ways:

**Method A: From Copilot Studio ("See agent in Teams")**
- Click "See agent in Teams" in Copilot Studio
- Opens the install experience directly in Teams
- Agent uses default name and icon from Copilot Studio settings
- No manifest customization in this method

**Method B: Manual Upload (Sideloading)**
- Download manifest .zip from Copilot Studio
- Customize manifest (name, icons, colors)
- Upload via Teams → Apps → "Upload a custom app"
- Requires sideloading to be enabled by admin policy
- **Recommended for DEV** with custom branding

**Characteristics:**
- No approval workflow
- Fastest deployment
- Best suited for testing and development
- Availability depends on tenant app permission policies
- Limited distribution (manual install or link sharing)

**Documentation:** [Upload custom apps](https://learn.microsoft.com/microsoftteams/platform/concepts/deploy-and-publish/apps-upload)

### Option 2: Maker Submission (Approval Workflow)

> **Best for:** Controlled rollout, QA validation, TEST environment
> **Governance Level:** Medium • **Speed:** Moderate • **Approval:** Admin required
{: .prompt-info }

A maker publishes the agent from Copilot Studio using **"Submit to your org"**.

**How It Works:**
1. In Copilot Studio → Channels → Microsoft Teams
2. Click **"Submit to your org"**
3. Creates a request for Teams admin (no immediate deployment)
4. Admin reviews and approves the app before it becomes available
5. Once approved, appears in org app catalog

**Characteristics:**
- Formal approval workflow
- App properties (name, icons, colors) configured in Copilot Studio
- Maker-initiated, admin-approved
- No direct editing of .zip manifest in this flow
- Still requires users to manually install from catalog (unless admin configures auto-install)

This is the standard governed path for maker-led deployments. It provides a balance between maker autonomy and administrative oversight.

**Documentation:** [Show agent in Teams app store](https://learn.microsoft.com/microsoft-copilot-studio/publication-add-bot-to-microsoft-teams#show-an-agent-in-the-teams-app-store-or-in-the-microsoft-365-agent-store)

### Option 3: Admin-Led Deployment (Manual Upload)

> **Best for:** Production rollouts, maximum control, auto-install + pinning
> **Governance Level:** High • **Speed:** Slower • **Approval:** Admin controlled
{: .prompt-tip }

This is the hero deployment path for production environments. A Teams admin uploads and deploys the app directly via the [Teams Admin Center](https://admin.teams.microsoft.com), unlocking powerful distribution features.

**How It Works:**
1. Maker downloads the Teams app manifest (.zip) from Copilot Studio
2. **Optional:** Modify the manifest before upload (name, icons, colors, version)
3. Admin uploads to Teams Admin Center → Manage apps

**Admin Capabilities:**
- Deploy org-wide or to specific security groups
- **Pin the app** using setup policies (appears in sidebar automatically)
- **Auto-install** for target user groups (no manual action required)
- Full control over availability and deployment timing

![Admin uploading custom app](/assets/posts/copilot-studio-teams-deployment/image23.png)
*Teams Admin Center - Upload custom app dialog*

This is the most controlled and scalable option for production rollouts. The ability to auto-install and pin apps transforms the user experience—instead of asking users to search for and install your agent, it simply appears in their Teams sidebar, ready to use.

**Documentation:**
- [Install agent in Teams](https://learn.microsoft.com/microsoft-copilot-studio/publication-add-bot-to-microsoft-teams#install-an-agent-in-teams-and-microsoft-365-copilot)
- [Teams custom app policies](https://learn.microsoft.com/microsoftteams/teams-custom-app-policies-and-settings)

## Manifest Customization for DEV and TEST

For DEV and TEST environments, you'll customize the Teams app manifest to differentiate from production. This prevents confusion and makes it immediately obvious which environment users are working with.

### Step 1: Download the Manifest

1. In Copilot Studio, go to **Settings → Channels**
2. Select **Microsoft Teams**
3. If not added, click "Add channel" and publish
4. Go to **Settings → Channels → Microsoft Teams → Availability options**
5. Click **"Download manifest"** (downloads a .zip file)

> The manifest .zip file is your deployment package. For DEV/TEST, you'll customize it before upload. For PROD with auto-install/pinning, use Option 3 (admin-led deployment).
{: .prompt-info }

### Step 2: Extract and Review

Extract the .zip file. You'll find three files:

```
manifest.zip/
├── manifest.json      # App configuration
├── color.png          # Full-color icon (192x192 px)
└── outline.png        # Outline icon (32x32 px)
```
{: .filepath}

### Step 3: Customize for Your Environment

The manifest.json file contains all the metadata Teams uses to display your agent. Here's what to customize:

![Manifest with DEV suffix](/assets/posts/copilot-studio-teams-deployment/image21.png)
*manifest.json showing "(DEV)" suffix and custom accent color*

```json
{
  "name": {
    "short": "B2E Agent (DEV)",
    "full": "B2E Agent - Development Environment"
  },
  "description": {
    "short": "Sample agent optimized for Teams",
    "full": "Agent instructions and topics are optimized for both Microsoft Teams and Microsoft 365 Copilot channels."
  },
  "accentColor": "#FFD700",
  "version": "1.0.2"
}
```

**Key changes:**
- Add `(DEV)` suffix to short name
- Custom accent color (yellow/gold for DEV)
- Incremented version number

### Color Psychology for Environments

Use distinct colors to create immediate visual differentiation:

| Environment | Accent Color | Color Meaning | Icon Treatment |
|-------------|--------------|---------------|----------------|
| **DEV** | `#FFD700` (Yellow/Gold) | Development, testing | Yellow tint or border |
| **TEST** | `#FFC300` (Amber) | Validation phase | Amber tint or border |
| **PROD** | `#0078D4` (Blue) or brand | Stable, production | Clean brand colors |

Users will instantly recognize which environment they're working in based on color alone.

### Step 4: Repackage

1. Select all three files (manifest.json, color.png, outline.png)
2. Compress to a new .zip file
3. **Important:** Files must be at the root level, not in a subfolder

> Files must be at root level of the .zip, not nested in a subfolder. Teams will reject improperly structured packages.
{: .prompt-warning }

**Verify structure:**
```
manifest_dev.zip/
├── manifest.json
├── color.png
└── outline.png
```
{: .filepath}

## Sharing Configuration in Copilot Studio

Before deploying, ensure your agent is shared with the right users in Copilot Studio.

![Copilot Studio sharing](/assets/posts/copilot-studio-teams-deployment/image22.png)
*Copilot Studio - Sharing agent with security group*

### Recommended: Use Security Groups

Create [Entra ID security groups](https://learn.microsoft.com/entra/fundamentals/how-to-manage-groups) for each environment:
- `B2E Agent Security Group (DEV)` - Makers, developers
- `B2E Agent Security Group (TEST)` - QA team, stakeholders
- `B2E Agent Security Group (PROD)` - All end users

### Configure in Copilot Studio

1. Settings → Security → Share
2. Select **Specific people or groups**
3. Add the appropriate security group
4. Set permission level ("Viewer" for end users)

> When publishing to the organizational app catalog, sharing must be set to **"Everyone in my organization"** for admins to see and approve the app. Admins can still limit exposure through Teams Admin Center and M365 Admin Center policies.
{: .prompt-tip }

## The Production Magic: Auto-Install and Pinning

Here's where Option 3 (Admin-Led Deployment) truly shines. Setup Policies in the Teams Admin Center enable **auto-install** and **pinning**, transforming your agent from "something users need to find" to "something that's already there."

### Why This Matters

Traditional app deployment requires:
1. Notify users about the new agent
2. Provide installation instructions
3. Hope they follow through
4. Field support tickets from users who can't find it

With auto-install and pinning:
1. Configure the policy once
2. Agent appears automatically for target users
3. No user action required
4. Immediate availability upon policy propagation

### Step 1: Configure Availability

Admin goes to Teams Admin Center:

![Edit availability](/assets/posts/copilot-studio-teams-deployment/image24.png)
*Teams Admin Center - Edit availability to target security groups*

**Availability Configuration:**
1. Teams Admin Center → Teams apps → Manage apps
2. Find and click the uploaded app
3. Click **"Users and groups"** tab or **"Edit availability"**
4. Select **"Specific users or groups"**
5. Add target security group (e.g., "B2E Agent Security Group (PROD)")
6. **Apply**

This controls who can see and install the app.

### Step 2: Create Setup Policy for Auto-Install and Pinning

This is the key step that enables automated deployment.

![Setup policy configuration](/assets/posts/copilot-studio-teams-deployment/image25.png)
*Setup policy showing auto-install (Installed apps) and pinning (Pinned apps) configuration*

**Setup Policy Creation:**
1. Go to **Teams apps → Setup policies**
2. Click **"Add"** to create new policy
3. **Policy name:** "B2E Agent Auto-Install"
4. **Installed apps section:**
   - Click **"Add apps"**
   - Search for "B2E Agent"
   - Select it → Click **Add**
   - This enables **auto-installation** (no user action required)
5. **Pinned apps section:**
   - Click **"Add apps"**
   - Search for "B2E Agent"
   - Select it and set position (e.g., position 5)
   - Click **Add**
   - This **pins the app in the sidebar**
6. **Save** the policy

> The "Installed apps" section auto-installs the agent. The "Pinned apps" section puts it in the sidebar at your specified position.
{: .prompt-tip }

### Step 3: Assign Policy to Target User Groups

![Assign policy to group](/assets/posts/copilot-studio-teams-deployment/image26.png)
*Assigning setup policy to B2E Agent Security Group (DEV)*

**Policy Assignment:**
1. Still in **Setup policies**, select your policy
2. Click **"Group policy assignment"** (recommended over individual assignment)
3. Select target security group (e.g., "B2E Agent Security Group (PROD)")
4. Set ranking (if multiple policies exist)
5. **Apply**

**Propagation:** May take up to 24 hours for full rollout. Users can sign out and back in to Teams to speed up installation.

### The User Experience

From the user's perspective:
1. User belongs to "B2E Agent Security Group (PROD)"
2. Policy propagates (or they sign out/in)
3. Teams client refreshes
4. Agent appears pinned in the sidebar at the configured position
5. **Zero user action required**

This is professional, enterprise-grade deployment. No training required, no support tickets, no confusion.

## Deploying DEV and TEST Environments

With production deployment covered, let's quickly address DEV and TEST.

### For DEV: Direct Install

**Quick deployment:**
1. Customize manifest (DEV suffix, yellow icons, `#FFD700` accent color)
2. Upload via Teams → Apps → "Upload a custom app"
3. Share link with other devs if needed

![DEV agent in Teams](/assets/posts/copilot-studio-teams-deployment/image2.png)
*B2E Agent (DEV) pinned in Teams sidebar with yellow visual treatment*

The yellow branding makes it immediately obvious this is the development agent. No risk of accidentally testing in production.

### For TEST: Maker Submission

1. Customize manifest for TEST (TEST suffix, amber icons, `#FFC300` accent color)
2. Submit via Copilot Studio → "Submit to your org"
3. Admin approves and assigns to TEST security group
4. Optional: Configure auto-install via Setup Policy (recommended for QA teams)

TEST gets the approval workflow but can still benefit from auto-install for your QA team.

Agents can be made available in [Microsoft 365 Copilot](https://learn.microsoft.com/microsoft-365-copilot/extensibility/overview-agents) alongside Teams.

![M365 Copilot availability](/assets/posts/copilot-studio-teams-deployment/image27.png)
*M365 Admin Center - Agent availability with specific users/groups option*

**M365 Copilot Deployment:**

1. **Access M365 Admin Center**
   - Work with a Microsoft 365 admin
   - Navigate to **Settings → Integrated apps → Agents**

2. **Configure Agent Availability**
   - Find your agent in the agent registry
   - Click to open agent details
   - Go to **"Available to"** tab
   - Select **Specific users/groups** (recommended)
   - Add target security group
   - **Update**

3. **Deploy and Pin (Optional)**
   - **Deploy** option: Auto-deploy to target users
   - **Pin for users** option: Agent appears in M365 Copilot's agent picker

The M365 Admin Center offers similar distribution controls to Teams, ensuring consistent governance across both channels.

## Updating Agents: Runtime vs. Manifest Changes

Understanding update types is crucial for efficient operations.

### Runtime Updates (No Manifest Changes)

**What:** Changes to agent behavior (orchestration, instructions, knowledge, actions, topics)

**Process:**
1. Make changes in Copilot Studio
2. Click **Publish**
3. Changes push automatically to all deployed channels
4. **No redeployment needed**

**Timeline:** Updates appear within minutes

This is the beauty of Copilot Studio's architecture. Behavioral changes deploy instantly across all environments without touching manifests or admin centers.

### Manifest Updates (Name, Icon, Description, Version)

**What:** Changes to how the app appears in Teams

**Process:**
1. Update `manifest.json` and **increment version number**
2. Update icons if needed
3. Repackage as .zip
4. Upload to Teams Admin Center as new version
5. Teams recognizes higher version number as an upgrade

> Version number must be higher for Teams to recognize it as an upgrade. Use semantic versioning: 1.0.0 → 1.0.1 (patch), 1.1.0 (minor), 2.0.0 (major).
{: .prompt-warning }

## Manifest Customization Deep Dive

Let's examine practical manifest customization scenarios.

### Basic Customization Fields

```json
{
  "$schema": "https://developer.microsoft.com/json-schemas/teams/v1.16/MicrosoftTeams.schema.json",
  "manifestVersion": "1.16",
  "version": "1.0.2",
  "id": "UNIQUE-GUID-HERE",
  "packageName": "com.contoso.b2eagent",
  "developer": {
    "name": "Contoso IT",
    "websiteUrl": "https://contoso.com",
    "privacyUrl": "https://contoso.com/privacy",
    "termsOfUseUrl": "https://contoso.com/terms"
  },
  "name": {
    "short": "B2E Agent (DEV)",
    "full": "B2E Agent - Development Environment"
  },
  "description": {
    "short": "Internal support agent for employees",
    "full": "Provides automated assistance for common IT, HR, and facilities questions. Development environment for testing."
  },
  "accentColor": "#FFD700",
  "icons": {
    "color": "color.png",
    "outline": "outline.png"
  }
}
```

**Key fields:**
- **version**: Must increment for updates to be recognized
- **name.short**: Max 30 characters, appears in sidebar
- **name.full**: Max 100 characters, appears in app details
- **description.short**: Max 80 characters, appears in search results
- **description.full**: Max 4000 characters, detailed description
- **accentColor**: Hex color, affects Teams UI chrome

### Environment-Specific Examples

**DEV Manifest:**
```json
{
  "name": {
    "short": "B2E Agent (DEV)",
    "full": "B2E Agent - Development Environment"
  },
  "accentColor": "#FFD700",
  "version": "1.0.0"
}
```

**TEST Manifest:**
```json
{
  "name": {
    "short": "B2E Agent (TEST)",
    "full": "B2E Agent - Testing Environment"
  },
  "accentColor": "#FFC300",
  "version": "1.0.0"
}
```

**PROD Manifest:**
```json
{
  "name": {
    "short": "B2E Agent",
    "full": "B2E Agent - Employee Support"
  },
  "accentColor": "#0078D4",
  "version": "1.0.0"
}
```

Notice PROD has no suffix—it's the canonical version. DEV and TEST are clearly marked variants.

## Troubleshooting Common Issues

### Users Don't See App After Policy Assignment

**Symptoms:** Policy assigned, but users don't see the agent pinned.

**Solutions:**
- Ask user to sign out and back in to Teams (most common fix)
- Verify user is in assigned security group
- Check app availability settings in Teams Admin Center
- Check policy assignment status in Setup policies → Manage users
- Wait up to 24 hours for full propagation

> Signing out and back in forces Teams to pull fresh policies. This resolves 80% of "app not appearing" issues.
{: .prompt-tip }

### App Shows "Blocked"

**Symptoms:** App appears but shows "Blocked" status.

**Solution:**
1. Teams Admin Center → Teams apps → Manage apps
2. Search for your app
3. Click app → Change status to **"Allowed"**

Apps are sometimes blocked by default when uploaded. This is an explicit admin approval step.

### Manifest Upload Fails

**Symptoms:** Upload fails with validation error.

**Common causes:**
- Files not at root level of .zip
- Invalid JSON syntax
- Missing required icons
- Icon dimensions incorrect (must be 192x192 and 32x32)
- Required fields missing from manifest.json

**Solutions:**
- Validate JSON at jsonlint.com
- Verify file structure (no parent folder in .zip)
- Check icon dimensions and file formats
- Review manifest schema documentation

### Agent Works in Copilot Studio But Not Teams

**Symptoms:** Agent responds correctly in Copilot Studio test chat but fails or behaves differently in Teams.

**Common causes:**
- Channel-specific authentication issues
- Teams-specific features not configured (adaptive cards, task modules)
- User not in sharing security group in Copilot Studio

**Solutions:**
- Test in Teams web client vs desktop client
- Check Copilot Studio sharing settings
- Review Teams channel configuration in Copilot Studio
- Check for Teams-specific generative answers settings

## Decision Matrix for Deployment Paths

Use this table to choose the right deployment path for your scenario:

| Scenario | Environment | Deployment Path | Manifest Custom | Auto-Install | Pinning |
|----------|-------------|-----------------|-----------------|--------------|---------|
| Active development, frequent changes | DEV | Option 1: Direct install | Yes (DEV suffix, yellow) | No | No |
| QA testing, stakeholder review | TEST | Option 2: Maker submission | Yes (TEST suffix, amber) | Optional | No |
| Production rollout to all users | PROD | Option 3: Admin-led | Optional (clean branding) | **Yes** | **Yes** |
| Pilot to small group before org-wide | PROD | Option 3: Admin-led | No | Yes (limited group) | Yes |
| Emergency hotfix testing | TEST | Option 1: Direct install | Yes | No | No |

## Final Thoughts

Copilot Studio makes building conversational agents accessible, but production deployment requires strategy. The three deployment paths—Direct Install, Maker Submission, and Admin-Led—each serve distinct governance needs:

- **Option 1** gives makers speed and flexibility for development
- **Option 2** adds approval workflows for controlled releases
- **Option 3** unlocks professional, enterprise-grade distribution with auto-install and pinning

The real differentiator is Setup Policies in Option 3. When you can auto-install and pin agents directly in user sidebars, you transform adoption from "find and install this tool" to "here's a tool that's already installed for you."

Pair this with manifest customization for DEV/TEST environments, and you have a complete ALM strategy: clear environment differentiation during development, governed release processes, and zero-friction production deployment.
